# Scaling Development from Two Contributors to Thousands

**Status:** Operational scaling guide

## Stage A — Founder plus one contributor

- all canonical changes use proposals;
- the non-author reviews where qualified;
- founder remains bootstrap canonicalization authority;
- sensitive work seeks external review rather than pretending two people provide broad independence;
- AI roles are separated into author, adversary, tester, and summarizer where practical.

## Stage B — Two to five regular contributors

- use protected integration branches or Forge integration objects;
- cross-review each other's exact heads;
- require founder/third reviewer for protocol-critical changes when two independent approvals are required;
- begin domain labels, reviewer registry, expiring delegations, and conflict declarations;
- never lower a constitutional threshold merely to make the counter convenient without recording the bootstrap exception.

## Stage C — Five to twenty contributors

- create incubating working groups around real workload, not titles;
- assign domain reviewers through explicit delegations;
- establish an integration council;
- use merge queues or deterministic integration objects;
- rotate review duty and track bottlenecks;
- create private security handling and audit coordination.

## Stage D — Twenty to one hundred contributors

- working groups become active with charters and terms;
- no founder review of every ordinary change;
- founder focuses on constitutional boundaries and transition evidence;
- use risk-based review policies and automated routing;
- publish authority, dependency, and integration maps;
- create contributor bounty pools with privacy-preserving claims;
- measure review quality, reversions, and unresolved integration debt.

## Stage E — One hundred to one thousand contributors

- governance operates through signed Forge objects;
- GitHub is a mirror or gateway;
- maintainers rotate and expire;
- working groups split when scope becomes unreviewable;
- cross-group changes require integration evidence;
- release authority is thresholded and separate from treasury;
- anonymous contribution remains open while scarce authority requires persistent cryptographic continuity;
- network governance begins replacing founder bootstrap authority only after personhood/legitimacy claims are honest and tested.

## Stage F — Network-native development

- proposals, reviews, approvals, provenance, bounties, disputes, releases, and delegations are Forge-native;
- canonical history does not depend on GitHub;
- owner adoption remains voluntary;
- governance can continue if founder, platform, or major working group disappears;
- GitHub may be read-only, reconstructed from Forge, or disabled through an accepted transition decision.

## Universal rules

At every scale:

- code authorship is not approval;
- AI output is not authority;
- payment is not governance weight;
- public identity is optional unless a narrowly scoped role legitimately requires disclosure;
- reviews bind to exact states;
- authority expires and is revocable;
- integration evidence is distinct from feature evidence;
- release does not force adoption;
- forks remain possible.
