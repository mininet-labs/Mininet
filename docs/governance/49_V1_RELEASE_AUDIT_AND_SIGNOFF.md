# Governance v1 Release Audit and Sign-off

**Status:** Normative release checklist

**Version:** 1.0

## Normative interpretation

This document is subordinate to `docs/FOUNDER_DIRECTIVES.md`, frozen invariants, and accepted decisions in the canonical Mininet repository. Where this package differs from canonical project authority, the canonical repository prevails until a governed amendment explicitly adopts the new rule.

Identity disclosure is never presumed. Anonymous, pseudonymous, and public participation remain valid unless a narrowly scoped role requires continuity or delegated authority. Persistent cryptographic accountability does not imply legal-name disclosure.

## Release audit

A v1 governance release candidate MUST be checked for:

### Hierarchy

- canonical founder directives and frozen invariants remain superior;
- no document grants itself authority;
- supersession and version rules are explicit.

### Sovereignty

- legal identity is not a general participation requirement;
- owner adoption remains voluntary;
- moderation and safety layers remain distinguishable from protocol validity;
- exit, export, and fork rights are preserved.

### Authority

- proposal, review, approval, canonicalization, release, adoption, and payment are distinct;
- authority is scoped, expiring, revocable, and exact-state bound;
- AI evidence cannot self-authorize;
- treasury authority cannot purchase governance authority.

### Continuity

- founder disappearance has a path;
- GitHub disappearance has a path;
- working-group succession has a path;
- key compromise and revocation have a path;
- minority fork and exit have a path.

### Honesty

- identity roots are not described as verified humans;
- signatures are not described as anonymity;
- unaudited cryptography remains experimental;
- specified mechanisms are not described as activated enforcement;
- open research remains explicit.

### Operability

- repository owner can deploy bootstrap controls;
- two contributors can work without self-approval;
- hundreds can coordinate through domains and integration;
- exceptions expire;
- machine-readable policy can be validated;
- conformance claims include evidence.

## Sign-off classes

- **Authorial sign-off:** confirms text is intentionally proposed.
- **Technical sign-off:** confirms consistency and implementability.
- **Security/privacy sign-off:** confirms claims match evidence.
- **Operational sign-off:** confirms current owner can execute bootstrap procedures.
- **Governance sign-off:** makes the exact package canonical.

No single sign-off substitutes for all others.
