# Governance Pack Changelog

## v1.1 — Standing AI engineering charter

- added a proposed, model-neutral Primary AI Engineer Charter for the bootstrap period;
- separated engineering stewardship and proactive in-scope execution from approval, quorum, canonicalization, release, administration, secret, treasury, and owner-adoption authority;
- made the Founder/Engineer relationship phase-scoped and subordinate to review, amendment, delegation, succession, and transition rules;
- added a repository-root `AGENTS.md` adapter intended to load the exact activated charter's Session Core at session entry;
- added an external activation record and schema binding charter, adapter, and summary digests without a self-hash cycle;
- added a canonical-checker `runtime` mode that rejects changed or newly introduced worktree instruction surfaces before candidate instructions are parsed;
- required activation against a separately supplied canonical checkpoint, a structured final Decision, an active matching phase, effective time, a truthful cooling basis, and no append-only supersession marker;
- added negative tests for branch self-activation, gate and loader drift, nested instruction injection, artifact drift, future activation, phase mismatch, incomplete cooling, supersession, and explicit AI Authority grants;
- added a machine-readable charter summary and named governance scenarios;
- protected the charter and session adapters through policy and review routing;
- added structural CI checks for missing, stale, or authority-expanding session adapters;
- mapped session guidance to a content-addressed Forge charter object; and
- preserved v1.0 as an immutable predecessor rather than silently replacing it.

## v0.3 — Governance protocol engineering foundation

- Added a layered architecture separating constitutional authority, ontology, state machines, operational policy, enforcement, and tests.
- Added the formal protocol ontology and reserved relation statements.
- Added normative drafting language and a standard governance-specification template.
- Added proposal, review, approval, integration, release, bounty, delegation, working-group, amendment, emergency, and GitHub-to-Forge state machines.
- Added a decision/authority table with identity modes, AI limits, independence requirements, and risk classes.
- Added a governance failure-mode catalogue covering founder loss, GitHub loss, Forge compromise, AI attacks, capture, treasury, releases, builders, personhood, forks, coercion, deadlock, and continuity drills.
- Added an executable-style governance scenario suite for contribution, review, integration, release, compensation, authority, continuity, constitutional integrity, and scaling.
- Added an experimental machine-readable summary schema and validation rules.
- Clarified that the existing concise vocabulary is subordinate to the fuller protocol ontology.

## v0.2 — Internal consistency and repository alignment

- Corrected normative precedence so the pack is explicitly subordinate to `SPEC-00`, frozen invariants, and accepted decision-log entries.
- Clarified that `01_DEVELOPMENT_CONSTITUTION.md` is a development-governance interpretation, not a competing canonical Constitution.
- Added one canonical vocabulary covering anonymous participation, pseudonymous continuity, AI review, human/pseudonymous approval, canonicalization, releases, and owner adoption.
- Added a cross-document consistency matrix linking principles to operations and expected enforcement.
- Added hard honesty rules for personhood, external audit status, Forge readiness, capability enforcement, and compensation authority.
- Standardized all document versions at 0.2.
- Replaced language that could imply current identity-root counting is verified-human governance.
- Preserved anonymous and pseudonymous contribution and compensation without making public legal identity compulsory.
- Clarified that responsibility may attach to a persistent pseudonymous governance identity.
- Clarified that GitHub remains operationally canonical until explicit Forge transition gates are satisfied.
- Clarified that platform procedures are evidence and implementation, not constitutional authority.
